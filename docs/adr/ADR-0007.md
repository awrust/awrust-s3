# ADR-0007: Use Behave and boto3 for BDD integration testing

## Status
Accepted

## Context
awrust-s3 is an S3 emulator. Its primary value proposition is compatibility with real S3 clients. Unit tests verify internal logic, but integration tests must prove that actual SDKs can talk to the server correctly.

We want:
- Tests that read like a compatibility specification
- Tests that use the same client libraries as real users
- Tests that run against both the binary and the Docker image
- Tests that cover both storage backends (memory and filesystem)

## Decision
Use **Behave** (Python BDD framework) with **boto3** (AWS SDK for Python) for integration testing. Gherkin feature files serve as a living compatibility spec.

## Alternatives considered
- **Rust integration tests with reqwest**: Fast, no Python dependency, but tests raw HTTP rather than SDK behavior. Misses SDK-specific quirks (e.g., how boto3 parses XML, multipart thresholds).
- **Rust integration tests with aws-sdk-s3**: Would test real SDK compatibility, but adds a heavy Rust dependency and slower compile times for tests.
- **Shell scripts with AWS CLI**: Fragile, hard to assert on, poor error reporting.
- **pytest with boto3**: Similar to Behave but without the Gherkin specification layer.

## Consequences
Positive:
- Feature files are readable by non-developers as compatibility documentation
- boto3 is the most widely used S3 client; if it works with boto3, it works for most users
- `environment.py` supports four modes: cargo/memory, cargo/fs, docker/memory, docker/fs
- Same scenarios test all backends and deployment modes
- Python test infrastructure is lightweight and fast to iterate

Negative:
- Requires Python + pip in CI (addressed via `actions/setup-python`)
- Separate language from the main codebase
- Behave step matchers require care to avoid ambiguity with similar patterns
