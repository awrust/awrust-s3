# ADR-0004: Use quick-xml for S3 XML serialization

## Status
Accepted

## Context
S3 API responses are XML: ListBucketResult, ListAllMyBucketsResult, Error responses, InitiateMultipartUploadResult, CompleteMultipartUploadResult, etc. We also need to parse XML request bodies (CompleteMultipartUpload).

We want:
- Serialize Rust structs to S3-compatible XML with minimal boilerplate
- Deserialize incoming XML request bodies
- Serde integration for consistency with the rest of the codebase

## Decision
Use **quick-xml** with the `serialize` and `serde` features for all S3 XML serialization and deserialization.

## Alternatives considered
- **xml-rs**: Mature but more verbose; lacks serde derive integration, requiring manual tree walking.
- **serde-xml-rs**: Built on xml-rs with serde support, but less maintained and slower than quick-xml.
- **Manual string formatting**: Used initially for error responses, but doesn't scale to complex nested structures like ListBucketResult with optional fields.
- **yaserde**: Serde-like but separate derive macros; smaller ecosystem and less adoption.

## Consequences
Positive:
- Serde derive attributes (`#[serde(rename = "...")]`) map directly to S3 XML element names
- Both serialization and deserialization from one crate
- Fast — zero-copy parsing, minimal allocations

Negative:
- Some S3 XML patterns (like wrapping Vec elements in parent containers) require intermediate structs
- `skip_serializing_if` needed for optional fields to match S3 response format exactly
