# ADR-0002: Use Axum as the HTTP framework

## Status
Accepted

## Context
awrust-s3 needs an HTTP routing layer capable of:
- Path-style routing: `/<bucket>` and `/<bucket>/<key...>`
- Method-based handlers: PUT/GET/HEAD/DELETE
- Query parsing for operations like ListObjectsV2
- Building responses with headers and optional XML bodies

We want:
- Low boilerplate for routing and request extraction
- A stable, widely adopted framework in the Rust ecosystem
- Minimal “magic” and easy-to-read handler code

## Decision
Use **Axum** as the HTTP framework for awrust-s3.

## Alternatives considered
- **Hyper directly**: Maximum control, but significantly more boilerplate (routing, extractors, error mapping).
- **Actix-web**: Mature and fast, but heavier framework with a different ecosystem style; adds more surface area than needed for v0.
- **Warp**: Good framework, but more functional/closure-heavy style can reduce readability for contributors unfamiliar with it.

## Consequences
Positive:
- Clean routing and extractors reduce boilerplate and speed up iteration
- Built on the Tokio + Hyper stack, which is widely supported
- Makes it easier to keep HTTP code thin and keep S3 semantics in the domain layer

Negative:
- Adds a framework dependency (vs using Hyper directly)
- Requires following Axum patterns for handlers and error handling

## Notes
Axum is chosen for v0 speed and clarity. If the project later needs features Axum cannot support well, we will revisit this decision with a measured migration plan.
