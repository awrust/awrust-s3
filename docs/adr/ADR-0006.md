# ADR-0006: Use Serde as the serialization framework

## Status
Accepted

## Context
The project serializes and deserializes data in multiple formats:
- XML for S3 API responses and request bodies (via quick-xml)
- JSON for structured logging (via tracing-subscriber)
- JSON for filesystem store metadata sidecar files (via serde_json)
- URL-encoded query parameters (via axum/serde_urlencoded)

A common serialization framework reduces boilerplate and keeps struct definitions consistent across all these formats.

## Decision
Use **Serde** (with `serde_json` for JSON and quick-xml's serde integration for XML) as the serialization framework across the entire project.

## Alternatives considered
- **Manual serialization**: More control, but massive boilerplate for every struct across every format.
- **Format-specific traits**: Would require separate derive/impl blocks per format; Serde unifies them.
- **prost / protobuf**: Not applicable; S3 uses XML and JSON, not protobuf.

## Consequences
Positive:
- Single `#[derive(Serialize, Deserialize)]` works for XML, JSON, and query params
- Rename attributes (`#[serde(rename = "...")]`) handle S3's PascalCase XML and kebab-case query params
- `skip_serializing_if` handles optional fields cleanly
- Widely adopted in Rust; contributors will already know it

Negative:
- Foundational dependency; appears in both crates
- Derive macros increase compile times
