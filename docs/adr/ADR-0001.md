# ADR-0001: Use Tokio as the async runtime

## Status
Accepted

## Context
awrust-s3 is an HTTP server that must handle multiple concurrent requests efficiently. The Rust ecosystem’s mainstream HTTP stack (Hyper-based frameworks) is async-first and typically relies on an async runtime to provide task scheduling, networking, and timers.

We want:
- Predictable concurrency handling (local dev + CI)
- A well-supported ecosystem for HTTP servers and async utilities
- A runtime that keeps future options open (e.g., async filesystem storage)

## Decision
Use **Tokio** as the async runtime for awrust-s3.

## Alternatives considered
- **async-std**: Smaller adoption in the ecosystem for server-side stacks; fewer “default” integrations.
- **Synchronous / blocking server**: Simpler initially, but worse scalability and would complicate future features (e.g., async IO, streaming).
- **Custom runtime / threadpool**: Reinvents critical infrastructure and increases maintenance risk.

## Consequences
Positive:
- Strong compatibility with modern HTTP frameworks and middleware
- Good performance and mature ecosystem
- Simplifies future async features (e.g., streaming bodies, async FS backend)

Negative:
- Foundational dependency; changing runtimes later would be expensive
- Requires async programming model throughout the server crate

## Notes
Tokio is treated as a foundational dependency and should not be replaced without strong justification and measured benefits.
