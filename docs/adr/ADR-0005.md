# ADR-0005: Use MD5 for ETag computation

## Status
Accepted

## Context
S3 ETags are MD5 hashes of object content for single-part uploads, and composite hashes for multipart uploads (`MD5(concat(part_md5s))-N`). ETags are returned on PUT, GET, HEAD, and list operations, and SDKs use them for integrity checks and conditional requests.

We must match S3's ETag format exactly for SDK compatibility.

## Decision
Use the **md-5** crate for all ETag computation, matching S3's behavior:
- Single-part: `"<hex(MD5(bytes))>"`
- Multipart: `"<hex(MD5(concat(MD5(part1), MD5(part2), ...)))>-<part_count>"`

## Alternatives considered
- **SHA-256 or other hash**: Would break SDK compatibility; S3 uses MD5 for ETags.
- **ring or rustcrypto digest**: More general crypto libraries, but md-5 is the focused, minimal choice for this single use case.
- **Skip ETag computation**: Not viable; SDKs validate ETags and `aws s3 cp` uses them for transfer verification.

## Consequences
Positive:
- Exact S3 compatibility for all SDK operations
- ETag computed once on put, stored with the object, no recomputation on read
- Composite ETag format works correctly with multipart uploads

Negative:
- MD5 is cryptographically broken; this is acceptable because ETags are not a security mechanism in our context (no auth in v0)
- md-5 crate is a dependency in both domain and server crates
