# ADR-0003: Use Tracing for structured logging and request instrumentation

## Status
Accepted

## Context
awrust-s3 is infrastructure software: when it fails or behaves unexpectedly, users need actionable diagnostics. We want to log:
- Request method/path
- Status code
- Latency
- Store backend (memory/fs)
- A request id to correlate logs

We also want to keep an upgrade path open for:
- JSON logs in CI environments
- Optional OpenTelemetry instrumentation
- Metrics integration

## Decision
Use **tracing** (and tracing-subscriber in the server crate) for logging and request-level instrumentation.

## Alternatives considered
- **println! debugging**: Not structured, not scalable, not suitable for OSS.
- **log + env_logger**: Fine for basic logs, but weaker request-scoped context and structured fields support compared to tracing.
- **slog**: Structured, but less “default” in modern Rust projects than tracing.

## Consequences
Positive:
- Structured logs with fields (better debugging and filtering)
- Request-level spans make it easy to correlate logs across a request
- Future-friendly: can add OpenTelemetry exporters without rewriting logging

Negative:
- Slightly more complex than basic logging initially
- Requires consistent usage patterns (fields/spans) to get full value

## Notes
Logging should remain minimal but useful. We will avoid overly verbose logs by default and rely on log levels for deeper diagnostics.
